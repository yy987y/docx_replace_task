# 幼儿园备课包 Word 自动替换项目交接文档

## 项目目标

这个项目用于处理幼儿园保教日志类 Word 模板。核心目标是：

- 读取带有文本框和图形时间轴的 `.docx`
- 识别每个活动块中的一级标签
- 仅替换二级标签和蓝色正文
- 保留一级标签、文本框结构、时间轴图形、字体样式和颜色

当前项目目录：`/Users/yanchao/Documents/develop/docx_replace_task`

## 当前文件说明

- `replace_word.py`
  当前主入口。负责读取 Word、定位活动段落、识别一级标签、调用本地生成器生成内容，并写出替换后的文档。
- `ai_generator.py`
  本地文案生成器。输入一级标签，输出二级标签和蓝色正文，不再依赖任何外部 LLM 接口。
- `replace_word_old.py`
  旧版替换逻辑。它更接近早期 memory 中记录的 run 级精细替换方式，可作为对照参考。
- `原文档.docx`
  当前完整测试输入文档。
- `保教日志_替换后.docx`
  最近一次运行的输出文档。
- `full_process.log`
  历史完整运行日志。
- `test_output.log`
  历史小规模测试日志。
- `tests/run_full_document_check.py`
  新增的完整文档自动化验证脚本，不修改业务代码，直接调用现有主流程。

## 文档结构约定

活动段落的逻辑结构是：

`一级标签：二级标签。蓝色正文`

其中：

- 一级标签保留不变
- 二级标签为黑色正文
- 蓝色正文为蓝色字体

当前 `replace_word.py` 识别的一级标签列表包括：

- 室内自主游戏
- 加点能量
- 早操律动
- 制作
- 主题活动
- 户外体育游戏
- 户外自主游戏
- 起床
- 户外自主游戏回应性照护
- 生活活动
- 抱抱离园
- 餐前活动
- 午餐
- 餐后活动

注意：

- 旧 memory 中曾使用过 `香香进餐`，当前主脚本里已变为 `午餐`
- `replace_word.py` 中的跳过列表仍是 `香香进餐`，如果后续发现跳过逻辑失效，需要优先检查这里

## 运行方式

### 1. 安装依赖

建议使用项目内虚拟环境：

```bash
python3 -m venv .venv
.venv/bin/python -m pip install Spire.Doc
```

### 2. 执行业务脚本

```bash
.venv/bin/python replace_word.py
```

默认读取 `原文档.docx`，输出 `保教日志_替换后.docx`。

### 3. 执行完整文档自动化验证

```bash
.venv/bin/python tests/run_full_document_check.py
```

这个脚本会：

- 调用现有 `process_document(...)` 跑本地生成 + 真实 Word 替换
- 输出完整运行日志到 `artifacts/full_document_test.log`
- 检查输出文档是否生成
- 检查输入输出活动段落数量是否一致
- 检查一级标签计数是否一致
- 检查一级标签后是否仍然存在冒号
- 检查替换后内容不是异常短文本
- 检查文档中不再出现 `幼儿进行活动（材料）。幼儿在活动。` 这类兜底占位句
- 检查至少有一个 `★户外自主游戏` 段落含有 `通过...帮助婴幼儿...` 的回应性照护文本

## 本次改造重点

- 移除了 `requests`、`API_URL`、`API_KEY` 以及所有外部网络依赖
- 生成逻辑改成“标签配置 + 材料池 + 动作模板 + 名字轮换”的本地组合式生成
- 替换逻辑改为按字符位置裁剪一级标签后的旧内容，解决“同一个 run 里混有一级标签、冒号和旧正文”时的残留问题
- 一级标签改为按长度倒序匹配，避免 `户外自主游戏` 抢先命中 `户外自主游戏回应性照护`

## 当前实现特点

### `replace_word.py`

当前替换策略是：

- 找到包含一级标签的段落
- 按字符位置截断，保留一级标签和原有冒号
- 删除其后的旧内容
- 再追加新的二级标签和蓝色正文
- 同时遍历 `Section.Body.Paragraphs` 和文本框中的段落

当前版本已经显著降低了“旧文字残留”和“冒号丢失”的概率；如果后续还要做更极致的格式保真，可以继续参考 `replace_word_old.py` 的 run 级思路。

### `ai_generator.py`

生成流程是本地两段式：

1. 先生成二级标签
2. 再基于二级标签生成蓝色正文

其中做了这些约束：

- 带 `★` 的一级标签要求更长正文
- `★户外自主游戏` 强制追加回应性照护句
- 同一标签最近场景和名字组合会做简单去重
- 蓝色正文会尽量沿用二级标签中的同一组材料，避免“标题和正文材料不一致”

## 已知风险

- 本地生成器当前是规则库方案，不是模型自由生成；稳定性高，但文案风格上限取决于模板池丰富度
- `香香进餐` 和 `午餐` 目前通过别名映射统一，但文档里两种叫法都可能出现
- 如果后续再新增一级标签，需要同时补充 `KNOWN_LEVEL1_TAGS` 和 `GENERATOR_CONFIG`
